# ReadAloud — Agent Instructions

## Repository Purpose

ReadAloud is a PDF/Document to Speech application. Users can upload documents (like PDFs), which are then parsed into books, chapters, and paragraphs. These can then be converted to high-quality speech using a custom TTS engine (ChatterBox).

## Architecture Overview

| Layer | Technology |
|---|---|
| Backend | .NET 10 ASP.NET Core — modular monolith |
| TTS Engine | Python FastAPI (ChatterBox) — CPU-based speech generation |
| Database | PostgreSQL 16+, EF Core + Npgsql |
| Frontend | React 19 + TypeScript + Vite, Tailwind CSS |
| State | Zustand |
| API clients | Auto-generated TypeScript via NSwag |
| Auth | JWT — Identity |
| Containers | Docker multi-stage builds, docker-compose for local |

## Backend Project Structure

Each module typically follows this layer structure:

```
src/backend/Modules/<Module>/
├── <Module>.Api/            — ASP.NET Core controllers, IModule DI registration
├── <Module>.Application/    — Business logic services
├── <Module>.Contracts/      — DTOs, enums, service interfaces
├── <Module>.Domain/         — EF Core entity classes
├── <Module>.Infrastructure/ — DbContext, repository implementations, migrations
└── <Module>.Tests/          — Unit tests
```

### Dependency direction

```
Api → Application → Contracts
Api → Infrastructure  (DI wiring only)
Infrastructure → Domain → Contracts
Tests → Application, Contracts, Domain
```

**Cross-module: only through `*.Contracts`.**

### Solution file

`src/ReadAloud.slnx` — all backend and frontend presentation projects. Use this path for all `dotnet` CLI commands.

## Forbidden Patterns

- **No hand-written migration files.** Always use `dotnet ef migrations add`.
- **No direct DbContext access across modules.**
- **No hand-editing `api-client.ts`.** Run `npm run generate`.
- **No manually defined TypeScript types that duplicate generated ones.**
- **No `.env` committed.**
- **No secrets or PII in logs.**
- **No unnecessary abstractions.**
